Column — Agentic AI and Product Value

Agentic AI時代にPSPA-Screenが
なお価値を持つ理由

Claude CodeやCodexを含むAgentic AIは、正直に言えば、公募案件の検索、要約、分類、比較、提案下書き、ワークフロー自動化のかなりの部分を実行できます。したがってPSPA-Screenの価値は、「AIにはできない」と主張することではありません。価値は、AIで可能な作業を、公共案件営業の反復可能で監査可能な業務プロセスへ圧縮することにあります。

公募案件インテリジェンスを、単発の調査・要約・提案書作成として見るなら、汎用Agentic AIで相当程度代替できます。公告ページを巡回し、PDFを読み、参加条件を抜き出し、Go／No-Goの仮説を出し、提案骨子を書くことは、もはや特殊な能力ではありません。

しかし、実務上の制約は別の場所にあります。案件判断には、自社の資格、地域、実績、体制、過去応募、証拠資料、開示可否、落札市場、締切、社内承認が絡みます。ここで必要なのは、うまい一回の回答ではなく、同じ前提で何度も判断でき、あとから根拠を追える仕組みです。

PSPA-Screenの価値は、Agentic AIの代替不能性ではない。
AIでできる作業を、公共案件営業の運用単位へ落とし込むことです。

Concession

まず、Agentic AIでも「できる」ことを認める

この点を曖昧にすると、訴求は弱くなります。Claude CodeやCodexに十分な指示、対象URL、企業情報、評価基準、過去実績、提出ルールを与えれば、PSPA-Screenに似た作業列を組むことは可能です。

汎用Agentが実行できること

  • 複数サイトの巡回と一覧化
  • 公告・仕様書・様式の要約
  • 参加資格、締切、提出物の抽出
  • 案件ごとの適合性スコア案
  • 提案書・質問票・社内メモの下書き

それでも残ること

  • 企業固有ファクトの正本管理
  • 未検証情報と提出可能情報の分離
  • 同じ評価軸による反復判断
  • 過去応募・落札市場との突合
  • 人間が承認できる証跡の保存

したがって、PSPA-Screenは「AIより賢い」と主張すべきではありません。むしろ、汎用AIをそのまま業務に入れた時に利用者側へ残る設定、検証、台帳化、再現性、レビュー負荷を引き受ける製品だと位置づける方が正確です。

Current implementation

現行実装が持つ差別点

PSPA-Screenの現行実装は、単なる「公告を読ませるプロンプト集」ではありません。製品コード、主体ごとの一次情報、派生設定、実行状態、提出前確認を分離する構造を持っています。

1. Product / Principal Pack / Compiler

CONTEXT

製品コードは主体固有の文字列を持たず、各社の identity、資格、対象機関、領域語、戦略条件は `principals/<id>/profile.yaml` に集約されます。`pspa-principal` がそこから keywords、triage、orgs、budget、scoring、eval_system を決定的に生成します。

2. 一次トリアージと4軸評価

SCREENING

取得した `Listing` は、まずLLMを使わないキーワード・期日ベースのトリアージに通されます。候補だけを feasibility、strategic fit、short-term cashflow、long-term fit の4軸で評価し、重みはprincipalごとに変更できます。

3. ファクト台帳と提出前ゲート

INTEGRITY

`fact_ledger.yaml` は、確認済み・要確認・矛盾ありの検証状態と、開示可否を別レイヤーで管理します。`PreSubmissionGate` は、ドラフト内の未検証ファクト、開示不可ファクト、個人名での参画約束を提出ブロッカーとして報告します。

4. 入札履歴・落札結果インテリジェンス

MEMORY

`bidding_ledger.yaml` は自社の応募・辞退・不採択・採択履歴を、`award_ledger.yaml` は他社を含む落札結果を扱います。落札結果側は `AwardRecord` として価格、予定価格、落札率、方式、入札者数、公告期間等を構造化し、価格妥当性や競合・パートナー候補の判断材料にします。

5. ローカルWebコンソールと読取専用ダッシュボード

OPERATIONS

Web UIは `127.0.0.1` 前提で、成果物閲覧、ジョブ起動、ケース確認を扱います。ダッシュボードはローカル成果物から再計算し、GET時にLLMやネットワークを呼びません。送信・自動応募のレバーは持ちません。

Why productization matters

自作できることと、運用できることは違う

Agentic AIで同等ワークフローを自作する場合、利用者は少なくとも次を設計しなければなりません。

✓

文脈境界：どの企業情報を正本とし、どのメモやドラフトを参考情報に落とすか。

✓

評価軸：資格適合、戦略親和性、短期収益、長期価値を毎回同じ意味で採点する方法。

✓

証拠管理：提案書に書いてよい実績、書いてはいけない実績、確認待ちの実績を分ける方法。

✓

差分処理：既読案件、再掲案件、資料受領後の再評価、落札結果の更新を扱う方法。

✓

セキュリティ：認証情報、業者コード、法人番号、証明書番号等を出力やログへ漏らさない方法。

✓

監査性：なぜその案件を追う／見送る判断をしたのか、あとから説明できる記録。

これらは、個別には実装可能です。しかし、公募営業の担当者が毎週使うには、設定、例外処理、権限、出力契約、テスト、ドキュメントが必要です。ここで製品の価値が出ます。

汎用Agentは作業を実行できる。PSPA-Screenは、その作業を反復可能な社内プロセスとして固定する。

Governance gap

公共案件では、生成物よりも根拠の管理が重い

公共調達は、単に魅力的な提案文を作ればよい領域ではありません。参加資格、証明資料、仕様適合、契約条件、開示可能な実績、価格根拠、過去応募との整合性が問われます。

AI利用に関する公的ガイダンスも、単なるツール選定ではなく、導入、文書化、透明性、リスク管理を重視しています。英国政府のAI PlaybookやAlgorithmic Transparency Recording Standard、EU AI Officeの説明は、AIの利用を制度・説明責任・監督の文脈で扱っています。[4][5][6]

これは発注者側AIに限った話ではありません。応札者側でも、提出物に虚偽・未検証・守秘未確認の事実が混入すれば、営業効率化の利得はすぐに消えます。

PSPA-Screenの二大原則は、「書いてある ≠ 検証済み」「提出した ≠ 証拠あり」です。これは単なる注意書きではなく、fact ledger、bidding ledger、pre-submission gate という実装に落ちています。

Positioning

PSPA-Screenは「AI代替不可」ではなく「実装圧縮」で売る

最も防御しやすい訴求は、次の形です。

Agentic AIは、公募案件インテリジェンスの成果物を生成するコストを下げる。
PSPA-Screenは、その成果物を実務で使える状態にするコストを下げる。

ここでいう実務で使える状態とは、案件を見つけることだけではありません。自社の資格・実績・体制に照らして評価し、証拠のある事実だけを外部提出物に使い、判断履歴を残し、落札結果から価格と競争環境を学習し、次回のGo／No-Goへ戻す状態です。

汎用Agentに残りやすい負担

  • 毎回の文脈投入
  • プロンプトと評価軸の属人化
  • 未検証ファクトの混入
  • ログ・台帳・証跡の分散
  • ツール連携の保守

PSPA-Screenが製品化する層

  • principalごとのSSOT
  • 決定的な派生設定生成
  • LLM前のルールトリアージ
  • 提出前ファクトゲート
  • 応募・落札履歴の再利用
Limits

過剰に言わない方がよいこと

PSPA-Screenは、現時点で全ての公共調達を網羅するものではありません。対象はアダプタ実装とprincipal設定に依存します。機関サイトの構造変更、認証ポータル、メール請求、PDF品質、OCR、LLM費用、法務上のスクレイピング許容範囲は、継続的な保守対象です。

また、スコアは判断支援であって、受注可能性や法的適合性を保証しません。落札結果の「1者応札」「短公告期間」「随契」等も、確定的な不正シグナルではなく、未検証の分析として扱う必要があります。

訴求上の境界

「AIにはできない」ではなく、「AIで作れるが、業務として安全に回すには実装・設定・検証・保守が必要で、その部分を製品化している」と表現するのが妥当です。

Conclusion

価値は、賢い回答ではなく、判断の再利用にある

公募案件営業では、毎回新しい案件を読む一方で、判断に使う企業側の事実、資格、体制、過去実績、証拠資料は繰り返し使われます。汎用Agentic AIは、その都度の読解と生成を速くします。しかし、再利用されるべき文脈を正本化し、未検証情報を止め、判断履歴を次に戻す仕組みまでは、利用者が設計しなければなりません。

PSPA-Screenが提供するのは、その設計済みの運用層です。公告取得、ルールトリアージ、4軸評価、Principal Pack、Compiler、ファクト台帳、提出前ゲート、落札結果インテリジェンス、Webコンソールを組み合わせ、公共案件営業を単発のAI作業から反復可能な業務へ移す。

したがって、Agentic AI時代にPSPA-Screenが持つ価値は、AIを否定することではありません。AIの能力を前提に、その能力を公共案件営業で使える形に固定することです。
